# Code Review

Review the change specified by `$ARGUMENTS`:
- A commit hash or range — review those commits.
- `staged` — review staged + unstaged changes against `HEAD` (used by `/ship`).
- No argument — review the latest commit (`HEAD`).

## Setup — gather real context first

- Read the full diff. Then read each changed file in full, not just the hunks.
- Glance at the surrounding directory to see how existing pages, components, or content entries handle the same concern.
- If the change touches the homepage, the footer, or `astro.config.mjs`, sanity-check that internal links still resolve and the sidebar wiring still matches the on-disk content folders.

## What to evaluate

These are heuristics, not a checklist to mechanically run through. Use judgment — flag what actually matters for *this* change.

### Intent

- Does it accomplish what the message / spec says?
- Anything half-finished, dead, or TODO-left in a critical path?

### Content (MDX)

- Frontmatter fields are present and consistent with siblings: `title`, `description`, `sidebar.order`, `lastUpdated`. Missing `description` hurts search snippet quality and Pagefind weighting.
- `sidebar.order` values inside one category folder don't collide and aren't all `1`.
- `lastUpdated` is plausible (today's date or the date the article was actually written, not a placeholder from the future).
- No PHI placeholder content anywhere. No fake client names, fake DOBs, fake session notes, fake invoice line items with personal details. Article copy may say "we're writing this; email admin@fledgepractice.com" but must not invent identifying details.
- No filler placeholder text from the Starlight template ("Get started building your docs site with Starlight" etc.).
- Internal links use absolute Starlight paths (`/clients/adding-a-client/`), trailing slash included. Relative `./` links inside MDX are fragile across the build.
- `mailto:` links go to `admin@fledgepractice.com` (the canonical support address), not a stale or personal address.

### MDX imports and components

- Components imported from `@astrojs/starlight/components` are spelled correctly (`Card`, `CardGrid`, `LinkCard`, `Tabs`, `TabItem`, `Aside`, `Steps`). Misspellings build cleanly but render as plain text.
- Custom components used in MDX exist on disk and are exported the way they're imported.
- Don't import React/Vue/Svelte components in MDX without the matching Astro integration installed — the build will break.

### Astro config and sidebar wiring

- Every category folder under `src/content/docs/` has a matching sidebar entry in `astro.config.mjs`, and every sidebar entry points at a folder that actually exists.
- `autogenerate: { directory: '<slug>' }` slugs match the folder names exactly.
- The `Footer` (or other component) override path in `components: { ... }` resolves to a file that exports a default Astro component.

### Brand tokens

- `src/styles/custom.css` is still imported via `customCss` in `astro.config.mjs`.
- New CSS doesn't override `--sl-color-accent*` with values that drift from the Fledge palette in `fledge-practice-dashboard/src/index.css`. Pull values from there if you need a new shade rather than guessing.
- The display heading font (Crimson Pro) is still loaded for headings on the splash and article pages.

### Footer and navigation

- Every link in the custom `Footer.astro` resolves — internal links to articles that exist, external links to live URLs (`app.fledgepractice.com`, `fledgepractice.com/...`).
- The `mailto:` is `admin@fledgepractice.com`.
- Status link is intentionally a non-link "coming soon" affordance per the support-hub spec; if a change makes it a real link, the status page must actually exist.

### Privacy boundary

This site is **public** and **content-only**. There is no auth, no user data, no PHI. Flag any change that:

- Adds an analytics script that ships data off-domain (Plausible self-hosted is fine; Google Analytics, Meta Pixel, Hotjar, etc. are not).
- Adds a contact form that posts somewhere — the spec says contact stays as a `mailto:` until a help-desk tool is wired up.
- Adds any client-side state, cookie, or fetch call that wasn't there before.

### Build and deploy

- Anything new that the build needs (an env var, a secret, a third-party API key) is not committed in plaintext and is documented in the README.
- New routes don't conflict with the Starlight default routes. If you add a top-level `src/pages/` page, it shadows Starlight's autogenerated route at the same path.

### Duplication and architecture

- Repeated copy across articles that's begging for a shared component (e.g. the "we're writing this; email us" stub used in 8+ articles is fine; it's intentional and short. The same paragraph repeated three times in one article should be collapsed.)
- Responsibilities in the right layer (component vs. content vs. config).
- Deviations from established patterns in this repo: justified, or accidental?

## What NOT to do

- Don't nitpick style, formatting, or naming unless it actually hurts readability.
- Don't suggest changes that are purely preferential.
- Don't flag things that are already idiomatic to this repo.
- Don't suggest abstractions for things that appear once. Three similar lines beats a premature helper.
- Don't push for backwards-compat shims or feature flags that aren't warranted.
- If the change is clean, say so. Don't pad.

## Output format

1. **Summary** — One or two sentences: what the change does and whether it achieves its goal.
2. **Issues** — Ranked by severity. Each entry:
   - **File and line**: `path/to/file.mdx:42` (or `.astro`, `.ts`)
   - **Severity**: Error / Warning / Suggestion
   - **Issue**: what's wrong and why it matters here
   - **Fix**: concrete change

If there are no real issues, say `No blocking issues.` and stop.
